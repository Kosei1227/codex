# Product Reframe Reference

Use this reference when a user request may be a literal feature request hiding a better product outcome.

## Reframe Sequence

1. Name the user's stated request in one sentence.
2. Identify the underlying pain, workflow, user, and urgency.
3. Ask what success looks like in the user's world, not only in the UI or code.
4. Surface 2 to 3 possible framings:
   - Literal implementation.
   - Narrowest valuable wedge.
   - Higher-leverage product version.
5. State the tradeoff for each framing: time, risk, learning value, technical debt, and user impact.
6. Recommend one framing and explain why.
7. Convert accepted framing into acceptance criteria and non-goals.

## Use Judgment

Do not expand scope by default. Offer expansions only when they materially improve the intended outcome or prevent building the wrong thing. Keep the user's constraints authoritative.

## Output Shape

Use concise labeled bullets:

- Confirmed goal.
- Reframed opportunity.
- Recommended slice.
- Explicitly out of scope.
- Acceptance criteria.
- Validation evidence needed.
